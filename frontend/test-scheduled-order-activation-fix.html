🔧 预约单激活状态修复测试

🐛 问题发现

问题描述：
预约单激活后状态没有从 SCHEDULED 改为 PENDING，导致其他司机上线后查询不到这个预约单。

影响：
• 预约单激活后仍保持 SCHEDULED 状态
• 新上线的司机无法看到已激活的预约单
• 只有激活时在线的司机能收到通知
• 降低了预约单的派单成功率

📊 状态流转对比

❌ 修复前的状态流转

SCHEDULED
→
SCHEDULED
→
ASSIGNED
问题： 激活后仍然是 SCHEDULED 状态，新上线司机查询不到

✅ 修复后的状态流转

SCHEDULED
→
PENDING
→
ASSIGNED
正确： 激活后变为 PENDING 状态，所有司机都能查询到

🔍 代码修复对比

❌ 修复前的激活逻辑

try { // 开始分配司机 log.info("开始为预约单分配司机: orderId={}", orderId); orderDispatchService.dispatchOrder(orderId); // 设置超时取消任务（30分钟后） scheduleOrderTimeout(orderId); log.info("预约单激活成功: orderId={}", orderId); } catch (Exception e) { // 激活失败，取消订单 cancelScheduledOrder(orderId, "系统激活失败"); } // ❌ 问题：没有将状态从SCHEDULED改为PENDING

✅ 修复后的激活逻辑

try { // ✅ 首先将预约单状态从SCHEDULED改为PENDING log.info("将预约单状态从SCHEDULED改为PENDING: orderId={}", orderId); order.setStatus("PENDING"); order.setUpdatedAt(LocalDateTime.now()); orderMapper.updateById(order); // ✅ 通知乘客预约单已激活 webSocketNotificationService.notifyPassengerOrderStatusChange( order.getPassengerId(), orderId, "PENDING", "预约时间已到，正在为您寻找司机..."); // 开始分配司机 orderDispatchService.dispatchOrder(orderId); // 设置超时取消任务 scheduleOrderTimeout(orderId); } catch (Exception e) { cancelScheduledOrder(orderId, "系统激活失败"); }

🔧 修复内容详解

1. 状态更新
• 在调用 dispatchOrder 之前先更新订单状态
• 从 SCHEDULED 改为 PENDING
• 更新 updatedAt 时间戳
2. 乘客通知
• 通知乘客预约单已激活
• 状态文本："预约时间已到，正在为您寻找司机..."
• 确保前端状态同步
3. 日志增强
• 详细记录状态变更过程
• 区分成功和失败的日志
• 便于问题排查和监控

• 激活前：status = 'SCHEDULED'
• 激活后：status = 'PENDING'
• updated_at 字段应该更新
前端状态检查：

• 司机端能查询到 PENDING 状态的预约单
