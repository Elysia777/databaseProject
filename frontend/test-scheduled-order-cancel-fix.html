🔧 预约单取消通知修复测试

🐛 问题发现

现象：
实时单的取消通知修复后正常工作，但预约单仍然存在同样的问题：司机重新登录后收不到预约单的取消通知。

根本原因：
预约单和实时单使用不同的取消逻辑：
• 实时单：OrderController.cancelOrderByPassenger() ✅ 有WebSocket通知
• 预约单：ScheduledOrderService.cancelScheduledOrder() ❌ 没有WebSocket通知

🔍 问题分析

❌ 修复前的预约单取消逻辑

public void cancelScheduledOrder(Long orderId, String reason) { Order order = orderMapper.selectById(orderId); order.setStatus("CANCELLED"); order.setCancelReason(reason); orderMapper.updateById(order); // ❌ 没有WebSocket通知司机！ log.info("预约单取消成功: orderId={}", orderId); }

✅ 修复后的预约单取消逻辑

public void cancelScheduledOrder(Long orderId, String reason) { Order order = orderMapper.selectById(orderId); Long driverId = order.getDriverId(); String originalStatus = order.getStatus(); order.setStatus("CANCELLED"); order.setCancelReason(reason); orderMapper.updateById(order); // ✅ 添加WebSocket通知司机 if (driverId != null && ("ASSIGNED".equals(originalStatus) || "PICKUP".equals(originalStatus))) { webSocketNotificationService.notifyDriverOrderCancelled(driverId, orderId, reason); } // ✅ 通知乘客 webSocketNotificationService.notifyPassengerOrderStatusChange( order.getPassengerId(), orderId, "CANCELLED", "预约单已取消：" + reason); }

🔧 修复内容详解

1. 添加司机通知逻辑
• 检查订单是否已分配给司机（driverId != null）
• 检查订单状态是否需要通知（ASSIGNED 或 PICKUP）
• 调用 WebSocket 通知服务发送取消消息
2. 添加乘客通知逻辑
• 通知乘客预约单已取消
• 包含取消原因信息
• 确保前端状态同步
3. 增强日志记录
• 记录原始状态和司机ID
• 详细的通知发送日志
• 异常处理和错误日志

🧪 测试流程

预约单取消通知测试步骤

步骤1： 乘客创建预约单（状态：SCHEDULED）
步骤2： 系统激活预约单（状态：PENDING）
步骤3： 司机2接单（状态：ASSIGNED）
步骤4： 司机2退出登录
步骤5： 司机1登录（可选，测试会话冲突）
步骤6： 司机2重新登录
步骤7： 乘客取消预约单
预期结果： 司机2收到预约单取消通知 ✅

🔍 关键修复点对比

实时单取消（已修复）：
• 接口：POST /api/orders/{id}/cancel
• 处理：OrderController.cancelOrderByPassenger()
• 通知：✅ 调用 webSocketNotificationService.notifyDriverOrderCancelled()
预约单取消（现已修复）：
• 接口：POST /api/orders/{id}/cancel-scheduled
• 处理：ScheduledOrderService.cancelScheduledOrder()
• 通知：✅ 现在也调用 webSocketNotificationService.notifyDriverOrderCancelled()
